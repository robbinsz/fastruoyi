## 第一章  项目概述

### 1.1 项目背景与目标

本项目是在开源框架 RuoYi-Vue3-FastAPI 基础上进行二次开发的代理分销管理系统，集成了**严格四级代理树结构**、**有时效性投注链接机制**和**多级收益计算模型**。

**核心目标：**

- 超级管理员分配第一个用户为总代理（顶级代理），享有管理员级别权限
- 支持**四级代理商树形结构**（总代理 → 高级代理 → 中级代理 → 初级代理），四级以外统一为普通客户
- 每级代理**默认不能设置次级代理**，此权限须由超级管理员显式开启
- 实现严格的数据隔离与界面隔离：每级代理只能看自己的直属下级代理和直属客户
- 总代理可生成多个**有时效性的投注链接**，用户点击确认后不可更改
- 总代理手动确认投注是否中奖，系统自动计算各级代理收益
- 提供代理商专属报表看板，支持多维度数据查询与导出

## 第二章  系统架构

### 2.1 整体架构图（文字描述）

```
【浏览器 / 客户端】
    Vue3 + Element Plus SPA
    根据用户角色（超级管理员 / 四级代理 / 普通客户）动态渲染菜单和页面

【后端 FastAPI 服务】
    路由层 (router)
        → 中间件：JWT 鉴权 + 层级权限校验 + 数据范围过滤
        → 业务层 (service)
        → 数据访问层 (dao/crud)
        → MySQL

【基础设施】
    Redis：Token 黑名单 / 投注链接（带 TTL）/ 报表缓存
    MySQL：所有业务数据持久化
```

### 2.2 四级代理层级结构设计

#### 2.2.1 层级定义

| 层级编号 | 角色名称   | 角色键        | 说明                                                         |
| -------- | ---------- | ------------- | ------------------------------------------------------------ |
| 0        | 超级管理员 | `super_admin` | 若依原有 admin，不参与收益，可控制所有权限                   |
| 1        | 总代理     | `agent_l1`    | 由超管分配，享有管理员级别操作权，可生成投注链接，手动确认中奖 |
| 2        | 高级代理   | `agent_l2`    | 由总代理创建（需超管授权总代理可创建次级）                   |
| 3        | 中级代理   | `agent_l3`    | 由高级代理创建（需超管授权高级代理可创建次级）               |
| 4        | 初级代理   | `agent_l4`    | 由中级代理创建（需超管授权中级代理可创建次级）               |
| 5        | 普通客户   | `customer`    | 四级代理之外统一为客户，由任意代理商创建                     |

#### 2.2.2 默认权限规则

> ⚠️ **核心约束：每级代理商默认不能创建次级代理，此权限须由超级管理员显式授权。**

```
默认状态：
  总代理    → 只能创建「普通客户」
  高级代理  → 只能创建「普通客户」
  中级代理  → 只能创建「普通客户」
  初级代理  → 只能创建「普通客户」

超管授权后：
  总代理    → 可创建「高级代理」和「普通客户」
  高级代理  → 可创建「中级代理」和「普通客户」
  中级代理  → 可创建「初级代理」和「普通客户」
  初级代理  → 只能创建「普通客户」（四级是最低代理层，下面只有客户）
```

#### 2.2.3 数据可见性规则（严格隔离）

每级代理登录后，**只能看到自己的直属下级代理（仅一层）和自己的直属客户**。

| 当前登录者    | 可见数据范围                              | 不可见数据范围                                    |
| ------------- | ----------------------------------------- | ------------------------------------------------- |
| 超级管理员    | 所有数据                                  | 无限制                                            |
| 总代理 (L1)   | 自己 + 直属高级代理（若有授权）+ 直属客户 | 高级代理的客户/下级、上级（无）、同级其他总代理   |
| 高级代理 (L2) | 自己 + 直属中级代理（若有授权）+ 直属客户 | 中级代理的客户/下级、上级总代理、同级其他高级代理 |
| 中级代理 (L3) | 自己 + 直属初级代理（若有授权）+ 直属客户 | 初级代理的客户/下级、上级代理、同级代理           |
| 初级代理 (L4) | 自己 + 直属客户                           | 上级代理、同级代理、其他客户                      |
| 普通客户      | 仅自己                                    | 所有代理商和其他客户                              |

> ⚠️ **关键：下级代理商在上级视角中具有「代理商 + 客户」双重身份，但其子数据（自己的客户、更深层代理）绝对不向上暴露。可见性只穿透一层。**

### 2.3 用户与代理的绑定关系

在 `sys_user` 表增加扩展字段，记录每个用户的层级身份和归属关系：

```sql
ALTER TABLE `sys_user`
  ADD COLUMN `agent_level`      TINYINT     DEFAULT 0
    COMMENT '代理层级：0=非代理/客户, 1=总代理, 2=高级代理, 3=中级代理, 4=初级代理',
  ADD COLUMN `belong_agent_id`  BIGINT      DEFAULT NULL
    COMMENT '归属代理商的agent_id：客户填直属代理商，代理商填上级代理商，顶级代理为NULL',
  ADD COLUMN `can_create_sub_agent` TINYINT DEFAULT 0
    COMMENT '是否有权创建次级代理：0=否（默认）1=是，由超管授权';
```

### 3.3 详细表结构 DDL

#### 3.3.1 agent_info（代理商扩展信息表）

```sql
CREATE TABLE `agent_info` (
  `agent_id`            BIGINT        NOT NULL AUTO_INCREMENT COMMENT '代理商ID',
  `user_id`             BIGINT        NOT NULL COMMENT '关联 sys_user.user_id',
```

#### 3.3.2 bet_link（投注链接表）

> 此表是本系统区别于普通分销系统的核心表，仅总代理可生成投注链接。

```sql
CREATE TABLE `bet_link` (
  `link_id`       BIGINT        NOT NULL AUTO_INCREMENT COMMENT '投注链接ID',
  `link_token`    VARCHAR(64)   NOT NULL COMMENT '链接唯一Token，UUID生成，用于URL拼接',
  `agent_id`      BIGINT        NOT NULL COMMENT '创建此链接的总代理agent_id',
  `link_name`     VARCHAR(100)  DEFAULT NULL COMMENT '链接备注名称，例如"3月1日第1期"',
  `bet_desc`      TEXT          DEFAULT NULL COMMENT '投注说明/描述，展示给用户看',
  `odds`          DECIMAL(6,2)  DEFAULT NULL COMMENT '本链接对应的赔率（总代理填写）',
  `expire_at`     DATETIME      NOT NULL COMMENT '链接过期时间，超时后用户无法访问',
  `max_users`     INT           DEFAULT NULL COMMENT '最大参与人数限制，NULL=不限',
  `status`        TINYINT       NOT NULL DEFAULT 0
    COMMENT '0=待投注 1=投注中 2=已截止 3=已确认中奖结果',
  `confirm_result` TINYINT      DEFAULT NULL COMMENT '总代理确认结果：0=未中 1=已中，NULL=未确认',
  `confirm_time`  DATETIME      DEFAULT NULL COMMENT '总代理确认时间',
  `create_time`   DATETIME      DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`link_id`),
  UNIQUE KEY `uk_token` (`link_token`),
  KEY `idx_agent_status`  (`agent_id`, `status`),
  KEY `idx_expire`        (`expire_at`)
) ENGINE=InnoDB COMMENT='投注链接（总代理生成，有时效性）';
```

#### 3.3.3 bet_record（投注记录表）

```sql
CREATE TABLE `bet_record` (
  `record_id`      BIGINT         NOT NULL AUTO_INCREMENT,
  `link_id`        BIGINT         NOT NULL COMMENT '关联投注链接',
  `user_id`        BIGINT         NOT NULL COMMENT '投注用户 sys_user.user_id',
  `belong_agent_id`BIGINT         NOT NULL COMMENT '用户的直属代理商agent_id',
  `bet_amount`     DECIMAL(12,2)  NOT NULL COMMENT '投注金额（用户确认时填写）',
  `odds`           DECIMAL(6,2)   NOT NULL COMMENT '投注时的赔率（从bet_link复制，确认后锁定）',
  `is_confirmed`   TINYINT        NOT NULL DEFAULT 0 COMMENT '用户是否已确认投注 0=未确认 1=已确认',
  `confirm_time`   DATETIME       DEFAULT NULL COMMENT '用户确认时间',
  `is_win`         TINYINT        DEFAULT NULL
    COMMENT '是否中奖：NULL=待总代理确认 0=未中 1=已中（总代理确认后写入）',
  `win_amount`     DECIMAL(12,2)  GENERATED ALWAYS AS
    (IF(is_win=1, bet_amount * odds, 0)) STORED COMMENT '中奖金额',
  `round_profit`   DECIMAL(12,2)  GENERATED ALWAYS AS
    (IF(is_win=1, bet_amount * odds - bet_amount, 0 - bet_amount)) STORED COMMENT '本轮收益',
  `create_time`    DATETIME       DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`record_id`),
  UNIQUE KEY `uk_link_user` (`link_id`, `user_id`) COMMENT '同一链接同一用户只能投注一次',
  KEY `idx_link`             (`link_id`),
  KEY `idx_user`             (`user_id`),
  KEY `idx_belong_agent`     (`belong_agent_id`)
) ENGINE=InnoDB COMMENT='投注记录（用户通过链接确认后锁定，不可更改）';
```

### 4.2 数据范围过滤中间件（核心）

所有涉及代理/客户查询的接口必须经过此中间件。

```python
# middleware/data_scope_middleware.py

class DataScope:
    type: str           # 'all' | 'agent' | 'self'
    agent_ids: list     # 可见的 agent_id 列表（含自己）
    user_ids:  list     # 可见的 user_id 列表

def get_visible_scope(current_user_id: int, db: Session) -> DataScope:
    user = get_user_by_id(current_user_id, db)

    if user.agent_level == 0 and is_super_admin(user):
        # 超级管理员：不过滤，返回全量
        return DataScope(type='all')

    if user.agent_level == 0:
        # 普通客户：只能看自己
        return DataScope(type='self', user_ids=[current_user_id])

    # 各级代理商（L1~L4）
    agent = get_agent_by_user_id(current_user_id, db)

    # 直属下级代理（仅一层，不递归）
    direct_sub_agents = get_direct_sub_agents(agent.agent_id, db)

    # 直属普通客户（belong_agent_id == agent.agent_id 且 agent_level=0）
    direct_customers = get_direct_customers(agent.agent_id, db)

    visible_agent_ids = [agent.agent_id] + [a.agent_id for a in direct_sub_agents]
    visible_user_ids  = (
        [current_user_id]
        + [a.user_id for a in direct_sub_agents]   # 下级代理的 user_id（双重身份）
        + [c.user_id for c in direct_customers]     # 直属普通客户
        # ❌ 绝对不包含 direct_sub_agents 各自名下的客户
    )

    return DataScope(
        type='agent',
        agent_ids=visible_agent_ids,
        user_ids=visible_user_ids
    )
```

### 4.7 收益计算引擎（核心逻辑）

**触发时机：** 总代理调用 `/bet/link/{link_id}/confirm-result` 确认中奖结果后，系统自动执行收益计算。

**计算步骤（严格按此逻辑实现）：**

```python
class EarningsCalculator:

    def calculate(self, link_id: int, db: Session):
        """
        总代理确认中奖后触发，按以下步骤在事务中完成计算
        """
        link = get_bet_link(link_id)
        confirmed_records = get_confirmed_records(link_id)  # is_confirmed=1 的记录

        with db.begin():
            for record in confirmed_records:
                # Step 1: 聚合单条投注记录数据
                bet_amount   = record.bet_amount
                win_amount   = record.win_amount   # 生成列，已自动计算
                user_profit  = record.round_profit  # 生成列，可为负

                # Step 2: 找到用户的直属代理商
                belong_agent = get_agent(record.belong_agent_id)

                # Step 3: 计算投注提成
                bet_commission = bet_amount * belong_agent.bet_commission_rate
                # 例：500 * 0.025 = 12.5

                # Step 4: 计算利润提成（查梯度表，个性化优先）
                if user_profit > 0:
                    config = query_commission_config(
                        agent_id=belong_agent.agent_id,
                        profit=user_profit
                    )
                    # 查询规则：先查 agent_id=belong_agent.agent_id，无则查 agent_id=NULL
                    profit_commission = config.commission_amt * config.split_ratio if config else 0
                else:
                    config = None
                    profit_commission = 0

                # Step 5: 汇总代理商总收益
                total_commission = bet_commission + profit_commission

                # Step 6: 写入用户收益快照
                insert user_earnings(
                    user_id=record.user_id,
                    link_id=link_id,
                    bet_amount=bet_amount,
                    win_amount=win_amount,
                    profit=user_profit
                )

                # Step 7: 写入直属代理商收益（earn_type='direct'）
                insert agent_earnings(
                    agent_id=belong_agent.agent_id,
                    source_user_id=record.user_id,
                    link_id=link_id,
                    bet_commission=bet_commission,
                    user_profit=user_profit,
                    profit_commission=profit_commission,
                    total_commission=total_commission,
                    earn_type='direct'
                )

                # Step 8: 上级代理分成（沿链路向上，直到L1或无上级）
                if config and belong_agent.parent_agent_id:
                    parent_commission = config.commission_amt * (1 - config.split_ratio)
                    insert agent_earnings(
                        agent_id=belong_agent.parent_agent_id,
                        source_user_id=record.user_id,
                        link_id=link_id,
                        profit_commission=parent_commission,
                        earn_type='inherit'
                    )

            # Step 9: 更新 bet_link 状态为已结算
            update bet_link SET status=3 WHERE link_id=link_id
```

> 💡 **注意**：当前设计上级分成只传递一层（直属上级）。如需多级向上传递分成，需在 Step 8 改为循环遍历代理链。请开发前与业务方确认传递层数。

## 第五章  投注链接机制（核心流程）

### 5.1 流程概述

```
总代理创建投注链接
    ↓
系统生成唯一 Token（UUID），存入 bet_link 表，同时在 Redis 设置 TTL
    ↓
总代理将链接（如 https://xxx.com/bet?token=xxxx）分发给用户
    ↓
用户点击链接 → 系统校验 Token 有效性和链接状态
    ↓
（有效）展示投注信息（赔率/说明/截止时间）→ 用户填写投注金额 → 点击「确认投注」
    ↓
系统写入 bet_record（is_confirmed=1，confirm_time=now()）→ 前端显示「投注已锁定，等待结果」
    ↓
总代理在后台看到所有已确认投注记录 → 手动确认是否中奖
    ↓
系统触发收益计算，写入 agent_earnings 和 user_earnings
```

### 5.3 创建投注链接的业务逻辑

```python
# POST /bet/link/create
# Request Body:
{
    "link_name": "3月1日第1期",
    "bet_desc":  "本期投注说明...",
    "odds":       3.0,           # 赔率
    "expire_at": "2026-03-01T20:00:00",  # 截止时间
    "max_users":  100            # 最多100人参与，NULL=不限
}

def create_bet_link(agent_id, data, db):
    # 1. 生成唯一 Token
    token = str(uuid.uuid4()).replace('-', '')

    # 2. 写入数据库
    link = BetLink(
        link_token=token,
        agent_id=agent_id,
        odds=data.odds,
        expire_at=data.expire_at,
        status=0
    )
    db.add(link)

    # 3. 写入 Redis（TTL = expire_at 距今秒数，用于快速校验）
    ttl = (data.expire_at - now()).total_seconds()
    redis.setex(f"bet_link:{token}", int(ttl), link.link_id)

    # 4. 返回完整链接 URL
    return f"https://yourdomain.com/bet/{token}"
```

### 5.4 用户点击链接的校验逻辑

```python
# GET /bet/open/{token}
def open_bet_link(token: str, current_user, db):
    # 1. 先查 Redis，快速判断是否过期
    link_id = redis.get(f"bet_link:{token}")
    if not link_id:
        raise HTTPException(410, "投注链接已过期或不存在")

    # 2. 查数据库
    link = get_link_by_token(token, db)
    if link.status == 2:
        raise HTTPException(410, "投注链接已截止")
    if link.expire_at < now():
        raise HTTPException(410, "投注链接已过期")
    if link.max_users and get_confirmed_count(link.link_id) >= link.max_users:
        raise HTTPException(409, "参与人数已达上限")

    # 3. 检查当前用户是否已投注（同一用户只能投注一次）
    existing = get_user_record(link.link_id, current_user.user_id, db)
    if existing and existing.is_confirmed == 1:
        return {"status": "already_confirmed", "record": existing}

    # 4. 返回链接信息供用户确认
    return {
        "link_name": link.link_name,
        "bet_desc":  link.bet_desc,
        "odds":      link.odds,
        "expire_at": link.expire_at,
        "status":    "open"
    }
```

### 5.5 用户确认投注的业务逻辑

```python
# POST /bet/open/{token}/confirm
# Request Body: { "bet_amount": 100 }

def confirm_bet(token: str, bet_amount: float, current_user, db):
    link = get_link_by_token(token, db)

    # 校验1：链接是否有效（同 open_bet_link 的校验）
    validate_link(link)

    # 校验2：同一用户同一链接只能确认一次
    existing = get_user_record(link.link_id, current_user.user_id, db)
    if existing and existing.is_confirmed == 1:
        raise HTTPException(409, "您已确认过此投注，不可重复提交")

    # 校验3：bet_amount 必须为正数
    assert bet_amount > 0, "投注金额必须大于0"

    # 写入投注记录（一旦写入 is_confirmed=1，永久锁定）
    record = BetRecord(
        link_id=link.link_id,
        user_id=current_user.user_id,
        belong_agent_id=current_user.belong_agent_id,
        bet_amount=bet_amount,
        odds=link.odds,           # 赔率从链接复制，确认后锁定
        is_confirmed=1,
        confirm_time=now()
    )
    db.add(record)

    # ⚠️ 此时 is_win=NULL，等待总代理确认
    return {"message": "投注已确认，请等待开奖结果"}
```

### 5.6 总代理确认中奖结果

```python
# PUT /bet/link/{link_id}/confirm-result
# Request Body: { "confirm_result": 1 }  # 0=未中 1=中奖

def confirm_result(link_id: int, confirm_result: int, current_agent, db):
    # 校验：只有 L1 总代理且是该链接的创建者才能确认
    link = get_link(link_id)
    assert link.agent_id == current_agent.agent_id, "无权操作此链接"
    assert link.status != 3, "此链接已确认结算，不可重复操作"

    with db.begin():
        # 1. 批量更新所有已确认记录的 is_win 字段
        update bet_record
        SET is_win = confirm_result
        WHERE link_id = link_id AND is_confirmed = 1

        # 2. 更新链接状态
        update bet_link
        SET status=3, confirm_result=confirm_result, confirm_time=now()
        WHERE link_id = link_id

        # 3. 触发收益计算
        EarningsCalculator().calculate(link_id, db)
```

### 8.3 接口鉴权规范

- 所有代理分销模块接口必须携带有效 JWT Token
- 后端每个接口在 Service 层调用 `get_visible_scope()` 获取数据范围，查询时带入 WHERE 条件
- **严禁前端传入 `agent_id` 参数直接查询** — 必须由后端根据 Token 中的 `user_id` 推导可见范围
- 创建下级代理时，后端必须同时校验：① `can_create_sub=1`；② `new_level <= 4`；③ `parent_agent_id == 当前代理的 agent_id`
- 总代理确认中奖时，后端必须校验 `bet_link.agent_id == 当前登录用户的 agent_id`
- 报表导出接口同样受 DataScope 约束，不得通过导出绕过数据隔离

### 8.4 投注锁定安全规范

- `bet_record.is_confirmed=1` 之后，该记录的 `bet_amount`、`odds` 字段设置数据库层面的更新保护（应用层拦截，禁止任何 UPDATE 操作）
- `bet_link.confirm_result` 一旦写入非 NULL，该链接的所有 `bet_record` 的 `is_win` 字段不允许再修改
- 前端确认弹窗必须包含「一旦确认不可更改」的明确提示文案

## 附录  关键公式与字段说明

### A. 核心计算公式汇总

| 字段                            | 公式                                 | 说明                                                   |
| ------------------------------- | ------------------------------------ | ------------------------------------------------------ |
| `win_amount`（中奖金额）        | `bet_amount × odds × is_win`         | `is_win=0` 时为0                                       |
| `round_profit`（本轮收益）      | `win_amount - bet_amount`            | 可为负值                                               |
| `bet_commission`（投注提成）    | `bet_amount × bet_commission_rate`   | 提成率在 `agent_info` 中配置                           |
| `profit_commission`（利润提成） | `commission_amt × split_ratio`       | `commission_amt` 查梯度表，仅 `round_profit > 0` 时有  |
| `total_commission`（总佣金）    | `bet_commission + profit_commission` | 代理商最终收益                                         |
| `parent_share`（上级代理分成）  | `commission_amt × (1 - split_ratio)` | 写入上级代理的 `agent_earnings`，`earn_type='inherit'` |

### B. 常见问题 Q&A

**Q1：默认每级代理不能创建次级代理，那第一个总代理怎么建立层级体系？**

超级管理员先通过「分配总代理」创建 L1，再通过「代理树管理」点击「授权创建次级」给 L1 开启权限，L1 才可以创建 L2。以此类推逐级授权。

**Q2：代理商 A 和代理商 B 是同级，A 能看到 B 吗？**

不能。无论同一层级还是跨层级，都是横向完全隔离的，代理商只能看到自己直属的下级。

**Q3：下级代理商在上级视角中是「代理商」还是「客户」？**

两者都是。在「我的下级代理」Tab 中以代理商身份出现，在「我的直属客户」Tab 中以客户身份出现（因为代理商本身也是上级的客户）。但其子数据（客户、更深层代理）不对上级可见。

**Q4：投注确认后用户后悔了可以撤销吗？**

不可以。`bet_record.is_confirmed=1` 写入后，前端不提供任何撤销入口，后端接口层面也拦截所有对该记录的修改请求。这是业务设计的强约束。

**Q5：总代理如果错误确认了中奖结果，可以修改吗？**

不可以。`bet_link.confirm_result` 一旦写入非 NULL 值，后端拦截所有修改请求，已计算的 `agent_earnings` 和 `user_earnings` 也不可回滚。这是业务设计的强约束。如需处理错误，建议超管介入后台直接修正数据，并记录操作日志。

**Q6：利润为负（用户亏损），代理商还有收益吗？**

有投注提成，但无利润提成。投注提成按总投注额计算，不论盈亏；利润提成只有在用户利润超过梯度最低门槛（默认100）时才产生。

**Q7：四级代理之外能再往下吗？**

不能。系统在后端硬性限制 `agent_level <= 4`，初级代理（L4）无法创建任何次级代理，只能创建普通客户。

**Q8：一个投注链接可以有多个用户确认投注吗？**

可以，只要未超过 `bet_link.max_users` 限制且链接未过期。每个用户只能对同一链接投注一次（`UNIQUE KEY uk_link_user`）。

**Q9：`belong_agent_id` 字段在哪里写入？**

两处写入：① 创建下级代理时，新代理的 `belong_agent_id = parent_agent_id`（直属上级的 `agent_id`）；② 创建普通客户时，新客户的 `belong_agent_id = 当前登录代理商的 agent_id`。此字段一旦写入不允许前端修改，只有 `super_admin` 可后台调整。
